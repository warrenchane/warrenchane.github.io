分布式运行环境

flink 运行时包含两类进程：

  • JobManagers(也称为master)协调分布式计算，他们负责调度任务、协调checkpoints、协调故障恢复
    每个Job至少有一个JobManger，高可用部署下会有多个JobManagers，其中一个作为leader，其余处于standby状态。
  • TaskManagers（也称为workers）执行dataflow中的tasks（准确来说是subtasks），并且缓存和交换数据streams
    每个Job至少会有一个TaskManager。

    导论

    Flink Application

    Stream：流，分为有限数据流和无限数据流。
    State：状态，是计算过程中的数据信息，在容错回复和CheckPoint中有很重要的作用。流计算本质上是Incremental Processing，因此需要不断查询保持状态。
    Time，分为Event time、Ingestion Time,Processing time,Flink的无限数据流是一个持续的过程，时间是我们判断业务状态是否滞后，数据处理是否及时的重要依据。
    API，API通常分为三层，由上而下可分为SQL/Table API、DataStream API、ProcessFunction三层，API的表达能力及业务抽象能力都非常强大，但越接近SQL层，表达能力会逐步减弱，抽象能力会逐步增强。越往下越灵活。

Flink Architecture

有界与无界数据流：一套框架能处理两种数据集合
部署灵活：支持Yarn、K8S
极高可伸缩性：峰值达17亿条/s
极致流式处理性能：本地状态存储，极致性能优化，减少了网络IO，但是在生产过程中，状态需要远程备份，因为本地存储是有可能会宕机导致状态丢失。

Flink Operation

  • 7 x 24小高可用：一致性与高效性CheckPoint
  • 业务应用监控运维：WebUI、Metric

有状态流式处理引擎

什么是有状态流式处理

传统批次处理方法：

  • 持续收取数据
  • 以时间作为划分数个批次档案的依据
  • 周期性执行批次运算

理想方法：

  • 累积状态

  • 时间：时间意味着引擎对于数据完整性有机制可以操控，当所有数据都完全接收到后，输出计算结果。

  • 产生精准的数据

Event-Time 处理

何时才能输出运算结果。
流式引擎是一直在运算的，需要有事件来告诉引擎是否已经完整收到运算所需要的数据。
Watermarks：是Flink中的特殊事件。

状态保存与迁移

流式应用是无时无刻都在运行的，运维上有几个重要的考量：

  • 改了bug，如何将之前的状态迁移到新的环境中来
  • 如何重新定义运行的平行化程度
  • 如何升级生产的版本

Savepoint （保存点）：一个手动产生的检查点（CheckPoint）
保存点记录着流式应用中所有的状态。

总结

  • 状态容错：精确一次保证，分布式快照
  • 可应付极大的状态量
  • 状态迁移：在应用重启时，可以恢复历史状态
  • Event-time处理：用以定义何时才算收完了资料，产生结果。

状态管理

什么是状态

需要使用状态的场景：

  • 去重
  • 窗口计算
  • 机器学习/深度学习
  • 访问历史数据

为什么需要管理状态

最直接的方式，是把所有的状态都放在内存里面，但是存储容量有限制，需要备份和恢复，我们可能还需要横向扩展。

理想的状态管理

  • 易用

Table API


Table API & SQL 的特性

  • 声明式——用户只关心做什么，不关心怎么做
  • 高性能——支持查询优化，可以获取更好的执行性能
  • 流批统一——相同的统计逻辑，既可以流模式运营，也可以批模式运行
  • 标准稳定——语义遵循SQL标准，不易变动
  • 易理解——所见即所得
